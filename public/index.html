Mastering the system by engineering the source.

I spent my time implementing the "hard parts"—consent logic, PKCE handshakes, and token rotation—from the ground up to truly understand how identity works.

The Sandbox

I built a demo client to showcase the full OIDC flow. It handles the handshake, pulls user claims, and manages sessions just like a production-ready application.

The Handshake

01. Initialization Client registers and hits the /authorization endpoint.
02. Identity Check Server verifies login status or prompts for credentials.
03. User Consent User approves scopes; server generates a secure one-time code.
04. The Exchange Client swaps code at /token for a full JWT payload.

Capabilities

  • [x] Authorization Code Flow
  • [x] PKCE / State / Nonce
  • [x] ID, Access & Refresh tokens
  • [x] Discovery & Userinfo
  • [x] Consent-based permissions

Internal Logic

Sessions are server-side and stored in Postgres. Each request creates a separate flow object, allowing a single user to have multiple client flows active in parallel without conflict.

User Control

Signed-in users can audit every app they've authorized and revoke access instantly. Revoking access removes both stored consent and all issued tokens for that app.

Behind the Engineering

"I wanted to see the edge cases—how state survives a redirect, where PKCE fits in the logic, and how parallel flows behave for the same user."

This project is a deep dive into the handshake protocols that power the modern web. It’s an exploration of identity, security, and how everything wires together when you build the provider yourself.
